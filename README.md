# Contact-Service-and-Unit-Tests

  My approach to designing software is a user-first approach. I begin by identifying the user's needs and the requirements established during the planning phase, then design each feature to satisfy those requirements while keeping the software solutions simple and reliable. Throughout development, I rely on requirements-based testing to verify that each functional requirement is supported by corresponding test cases. I also recognize that no amount of testing can prove software is completely free of defects, so I follow the heuristic principle by striving for strong test coverage, typically aiming for at least 80% code coverage in unit testing. I also keep in mind that thoughtful test design is just as important as the percentage itself. Using techniques such as branch testing also helps ensure that all unique decision paths within the code are exercised, increasing confidence that the application behaves correctly under a variety of conditions.

Working on the Contact Service project significantly reinforced the importance of selecting appropriate testing strategies for each phase of the SDLC. Early in development, unit testing allows me to verify that individual classes and methods behave correctly before integrating them into the larger application. As the project progressed, testing became an iterative process where each new feature or modification was followed by additional test cases to confirm existing functionality had not been broken. This experience solidified my understanding that software quality is not achieved through a single round of testing at the end of a project. Only through continuous verification and refinement throughout development will quality be achieved.
